# IoT Dashboard 系統操作指南

## 系統架構概覽

系統採用**全 UI 操作、DB 儲存**的彈性設計。所有產線、設備、感測器設定皆透過介面完成，不需修改程式碼。

| 資料類型 | 儲存位置 | 修改方式 |
|---------|---------|---------|
| 產線 / 設備配置 | 後端 DB | UI 操作 |
| 感測器屬性類型 | 後端 DB | UI → 屬性類型 |
| 設備連線（PLC/API） | 後端 DB | 設備整合精靈 |
| UCL / LCL 限值 | 後端 DB | UI → 限值設定 |
| 鞋子在位偵測 | 後端 DB | 綁定 `material_detect` 屬性 |

> **注意：** 目前前端仍保留一個舊的「烤箱生產線」Tab（硬編碼在 `liveLineConfig.ts`）。
> 正確做法是透過本指南的流程，用 UI 重新建立它，之後再從程式碼移除舊 Tab。

## 第一部分：建立新產線

### 1-1 新增產線

1. 點左上角**產線下拉選單**
2. 滾到底部 → **＋ 添加产线**
3. 輸入產線名稱（例如「烤箱生產線」）→ 按 Enter

### 1-2 刪除產線

1. 開啟**編輯模式**（右上角鎖頭圖示）
2. 點產線下拉 → 滑鼠移到產線名稱 → 出現垃圾桶圖示 → 點擊
3. 確認對話框會顯示產線名稱 → 點「刪除」確認

---

## 第二部分：新增設備到產線（設備整合精靈）

每台真實 PLC 設備都透過**設備整合精靈**建立，共 7 步。

### 開啟精靈

右上角「＋ 新增設備」下拉選單提供兩個選項：

| 選項 | 用途 | 步驟數 |
|------|------|--------|
| **整合新設備** | 從零開始：選協議、掃描、建立設備類型 | 7 步 |
| **使用已有模板** | 從已建立的設備類型新增卡片到產線 | 3 步 |

選擇「整合新設備」進入精靈：

### Step 1：選擇通訊協議

| 協議 | 適用情境 |
|------|---------|
| **Modbus TCP** | 標準 PLC（西門子、三菱、台達等） |
| **Web API** | 設備提供 REST API |
| **Push Ingest** | 設備主動推送（MQTT gateway 等） |

### Step 2：填入連線設定

**Modbus TCP：**
```
主機位址：設備 IP（例如 192.168.1.100）
Port：502
單元 ID：1（大多數設備預設）
Modbus 功能碼：holding（類比值）/ discrete（DI 開關）
```

**Web API：**
```
端點 URL：http://設備IP/api/data
HTTP 方法：GET
JSON 路徑：data.value（點號路徑）
輪詢間隔：5000（毫秒）
```

### Step 3：掃描設備資料點

點 **掃描設備** → 系統列出所有可用暫存器 → 勾選要監控的點。

### Step 4：選取資料點

從掃描結果中勾選要監控的暫存器。

### Step 5：貼屬性標籤

每個選取的資料點設定：
- **屬性類型**：溫度 / 壓力 / 濕度 / 流量 / 計數器 / 狀態 / 在位
- **顯示名稱**：自訂（例如「大底溫度」）
- **單位**：℃ / kPa / % 等

### Step 6：設定設備類型

```
設備類型名稱：例如「藥水箱」
顯示類型：
  single_kpi      — 單一大數字（1 個感測點）
  dual_side_spark — 左右兩組（2-3 個感測點）
  four_rings      — 四個環形指針（4 個感測點）
  molding_matrix  — 矩陣格式（6 個感測點）
```

### Step 7：確認建立

點 **建立** → 系統自動：
1. 建立 DeviceConnection（開始 Polling）
2. 建立 EquipmentType（設備類型範本）
3. SSE 推送更新到所有瀏覽器

### 精靈完成後：加入產線

精靈建立的是「設備類型範本」，還需手動加入產線：

1. 右上角下拉 → **加入既有類型**
2. 選取剛建立的設備類型
3. 完成 → 設備卡片出現在儀表板

## 第三部分：手動建立「烤箱生產線」— 完整步驟

從零開始把烤箱生產線建起來，含 12 個溫度感測器 + 32 路 DI 鞋子在位偵測。本段所有資料先列在前面，後面的 6 個步驟再回頭引用。

---

### 開始前：要準備的資料

#### 1. PLC 連線參數（來自 OvenDataReceive）

| 欄位 | 值 |
|------|---|
| 主機位址 | `192.168.62.74` |
| Port | `502` |
| Unit ID | `1`（溫度 + DI 共用） |
| 資產編號 (AssetCode) | `OVEN-LINE-01` |
| 序號 (SerialNumber) | `SHX32rDtIQc4ahtc` |

> **DI 與溫度共用同一台 PLC**，但精靈要建**兩條獨立連線**（因 Modbus 功能碼不同）。詳見下面「Modbus 功能碼」段。

#### 2. 溫度感測器對照（12 路，FC03 Holding Register，40001 起連續）

| Register | SensorId | 設備 | 測量點 | UCL | LCL | 單位 |
|----------|----------|------|--------|-----|-----|------|
| **40001** | 1 | 高速加熱定型機 | 設備溫度 | 185 | 130 | ℃ |
| **40002** | 2 | 藥水箱 | 大底溫度 | 85 | 55 | ℃ |
| **40003** | 3 | 藥水箱 | 鞋面溫度 | 80 | 50 | ℃ |
| **40004** | 4 | 一次膠 | 大底溫度 | 80 | 50 | ℃ |
| **40005** | 5 | 一次膠 | 鞋面溫度 | 75 | 45 | ℃ |
| **40006** | 6 | 二次膠 | 大底溫度 | 80 | 50 | ℃ |
| **40007** | 7 | 二次膠 | 鞋面溫度 | 75 | 45 | ℃ |
| **40008** | 8 | 冷凍機 | 設備溫度 | -10 | -30 | ℃ |
| **40009** | 9 | 後跟定型 | 熱定型右 | 135 | 100 | ℃ |
| **40010** | 10 | 後跟定型 | 冷定型右 | 20 | 3 | ℃ |
| **40011** | 11 | 後跟定型 | 熱定型左 | 135 | 100 | ℃ |
| **40012** | 12 | 後跟定型 | 冷定型左 | 20 | 3 | ℃ |

> **資料格式：** 16-bit unsigned，先 byte swap，再乘以 0.1。例：原始值 256 → byte swap → 0x0100 → 256 → ×0.1 → **25.6 °C**
>
> **RS485 Daisy Chain 注意：** 藥水箱（sensors 2、3）是 RS485 串聯鏈起點，若斷電或斷線，sensors 4-12 全部讀不到。

#### 3. 設備類型對照（共 7 個 EquipmentType — 6 溫度 + 1 DI）

每一列代表精靈跑一次：

| 設備類型名稱 | 顯示類型 | 起始位址 | 讀取數量 | 對應點位 | 功能碼 |
|-------------|---------|---------|---------|---------|--------|
| 高速加熱定型機 | `single_kpi` | 40001 | 1 | 40001 | `holding` |
| 藥水箱 | `dual_side_spark` | 40002 | 2 | 40002, 40003 | `holding` |
| 一次膠 | `dual_side_spark` | 40004 | 2 | 40004, 40005 | `holding` |
| 二次膠 | `dual_side_spark` | 40006 | 2 | 40006, 40007 | `holding` |
| 冷凍機 | `single_kpi` | 40008 | 1 | 40008 | `holding` |
| 後跟定型 | `four_rings` | 40009 | 4 | 40009-40012 | `holding` |
| **烤箱DI集中器** | `single_kpi` | **0** | **32** | **DI bit 0-31** | **`discrete`** |

#### 4. DI 配置（32 路鞋子在位偵測）

對應 OvenDataReceive 的 `ReadDiscreteInputs(unitId=1, startAddress=0, count=32)`，每個 bit 一個工位的光電開關。**並非 register 40013**（之前舊邏輯）。

---

### Modbus 功能碼說明

精靈 Step 2 有「**Modbus 功能碼**」下拉，這是分流溫度跟 DI 的關鍵：

| 選項 | Modbus FC | 訊號類型 | 數值 | 用途 |
|------|-----------|---------|------|------|
| `holding` | **FC03** Holding Register | 類比 / 數字 | 16-bit 整數 | 溫度、壓力、濕度、流量、轉速 |
| `discrete` | **FC02** Discrete Input | 數位 ON/OFF | 1-bit (0/1) | 光電開關、限位、**鞋子到位**、急停 |

選 `discrete` 時 `dataType` / `scale` / `byteSwap` 會被忽略，回值固定 `0.0` 或 `1.0`。

---

### 步驟 1：建立產線

1. 左上角產線下拉 → **＋ 添加產線**
2. 輸入 `烤箱生產線` → Enter

---

### 步驟 2：建立溫度連線（精靈跑 6 次，依「設備類型對照」表前 6 列）

對「設備類型對照」表的 **前 6 列**（高速加熱定型機 → 後跟定型）每一列**重複跑一次精靈**：

1. 右上角 → **＋ 新增設備** → **整合新設備**
2. **Step 1**：選 **Modbus TCP**
3. **Step 2** 連線設定：

   | 欄位 | 填入 |
   |------|------|
   | 主機位址 | `192.168.62.74` |
   | Port | `502` |
   | Unit ID | `1` |
   | 起始位址 | 依對照表（如藥水箱 `40002`） |
   | 讀取數量 | 依對照表（如藥水箱 `2`） |
   | 資料型別 | `uint16` |
   | Byte Swap | ✅ 開啟 |
   | 縮放係數 | `0.1` |
   | **Modbus 功能碼** | **`holding`**（預設值，不用改） |
   | 連線名稱 | 例如 `烤箱-藥水箱` |

4. **Step 3** 點「掃描設備」→ 看到該範圍的 register
5. **Step 4** 全選掃到的 register
6. **Step 5** 對每個 register 設定：
   - 屬性類型：**「溫度」**
   - 顯示名稱：依「溫度感測器對照」表（例如「鞋面溫度」、「大底溫度」）
   - 單位：`℃`
7. **Step 6** 設備類型：
   - 名稱：依對照表（如 `藥水箱`）
   - 顯示類型：依對照表（如 `dual_side_spark`）
8. **Step 7** 建立 → 跑下一個設備

---

### 步驟 3：建立 DI 連線（精靈跑 1 次）

1. 右上角 → **＋ 新增設備** → **整合新設備**
2. **Step 1**：選 **Modbus TCP**
3. **Step 2** 連線設定：

   | 欄位 | 填入 |
   |------|------|
   | 主機位址 | `192.168.62.74`（與溫度同 IP） |
   | Port | `502` |
   | Unit ID | `1` |
   | **起始位址** | **`0`** ← DI 從 0 開始，不是 40001 |
   | **讀取數量** | **`32`** |
   | **Modbus 功能碼** | **`discrete`** ← **重點，這就是 DI** |
   | 連線名稱 | `烤箱-DI` |
   | （`dataType` / `scale` / `byteSwap` 會被忽略） | — |

4. **Step 3** 掃描 → 看到 32 個 bit 點位（DI01 ~ DI32）
5. **Step 4** 勾選實際接到光電 / 到位開關的那幾個 bit
6. **Step 5** 屬性類型選 **「在位」**（material_detect）
7. **Step 6** 設備類型：名稱 `烤箱DI集中器`，顯示類型 `single_kpi`
8. **Step 7** 建立

---

### 步驟 4：把 7 條設備加入產線

> ⚠️ **DI 集中器一定要加入產線並綁 AssetCode！** 否則步驟 6 的 gating 來源下拉**找不到那些 DI bit**（後端 `/api/sensor-gating/candidates` 只列已加入 LineConfig 且有 AssetCode 的 sensor）。

1. 切到「烤箱生產線」Tab
2. 右上角 → **＋ 新增設備** → **加入既有類型**
3. 選剛建好的設備類型 → 加入產線
4. 設備綁定 AssetCode：
   - 6 個溫度設備：`OVEN-LINE-01`
   - 1 個 DI 集中器：`OVEN-LINE-01-DI`（建議獨立 AssetCode 方便辨識）
5. 重複到 7 個設備（6 溫度 + 1 DI）全部加完

---

### 步驟 5：設定 UCL / LCL 限值

1. 右上角 **🎚 限值設定**
2. 選 AssetCode `OVEN-LINE-01`
3. 依「溫度感測器對照」表填入每個 sensor 的 UCL / LCL
4. 儲存 → **即時生效**（alert 立刻按新閾值判斷）

---

### 步驟 6（選用）：用 DI 控制溫度採樣（Sensor Gating）

只有鞋子在工位才記錄溫度，避免空轉時段污染統計：

1. 在步驟 5 的限值 modal 裡，找到要 gating 的溫度 sensor（例如 `40003 鞋面溫度`）
2. 該列下方展開 **⚙ 條件採樣**
3. 勾選「啟用 gating」→ 來源下拉選 `烤箱-DI` 對應的 DI bit（**支援跨 AssetCode**）
4. （選用）調整 `DelayMs`（預設 0，gating 升起後多久才開始採樣）和 `MaxAgeMs`（預設 1000，DI 訊號過期視為未到位）
5. 儲存 → 下一個 polling tick 立刻生效

**三態效果：**

| DI 狀態 | 溫度行為 | Dashboard 顯示 |
|---------|---------|---------------|
| `1`（鞋子在位） | 照常記錄 | 🟢 採樣中 |
| `0`（無鞋子） | 完全不寫 DB | ⚪ 待機 |
| 超過 `MaxAgeMs` 沒更新 | fail-safe 視為未到位 | 🔴 異常 |

詳細設計：[`docs/superpowers/specs/2026-04-27-sensor-gating-design.md`](superpowers/specs/2026-04-27-sensor-gating-design.md)

---

### 驗證：應該看到什麼

完成後 dashboard：
- 「烤箱生產線」Tab 有 **7 張卡片**（6 溫度 + 1 DI 集中器）
- 各溫度卡片即時顯示數值，超出 UCL/LCL 變紅 / 觸發 alert
- DI 卡片顯示各 bit 的 ON/OFF 狀態
- 若有設 gating，溫度卡片右上有「採樣中／待機／異常」三態 badge

連線健康：右上角 **⚙️ 系統設定 → 連線管理** → 看到 7 條 connection 全 healthy（0 連續錯誤）。

## 第五部分：屬性類型管理

### 系統內建 8 個屬性類型（不可刪除）

| 名稱 | Behavior | 預設單位 | 說明 |
|------|---------|---------|------|
| 溫度 | normal | ℃ | 一般量測值 |
| 壓力 | normal | kPa | 一般量測值 |
| 濕度 | normal | % | 一般量測值 |
| 流量 | normal | L/min | 一般量測值 |
| 計數器 | counter | count | 累計計數 |
| 狀態 | state | — | 開關狀態 |
| 資產編號 | asset_code | — | 設備識別碼 |
| 在位 | material_detect | — | 鞋子在位偵測 |

> **在位偵測說明：** 只要某個感測點綁定了「在位」屬性類型，系統自動識別為在位感測器（原 PLC register 40013），不需硬編碼 SensorId。

### 新增自訂屬性類型

右上角 **⚙️ 系統設定** → **屬性管理** → **新增** → 填入名稱、圖示、預設單位、UCL/LCL 預設值

新增後在精靈 Step 5 的下拉清單中出現。

## 第六部分：系統監控

### 查看設備連線健康狀態

右上角 **⚙️ 系統設定** → **連線管理** → 顯示所有 DeviceConnection 狀態

或直接查 API：
```
GET /api/diagnostics/polling
```

回傳每個設備的：連線狀態（healthy / error / disabled）、連續錯誤次數、最後 Poll 時間、錯誤訊息

### Circuit Breaker 機制

- 連續 **3 次** Poll 失敗 → 切換慢速重試（每 30 秒一次）
- 恢復成功後自動回到正常間隔

### SSE 即時推送

設備連線、屬性類型等設定變更 → 自動推送到所有已開啟的瀏覽器，無需手動重新整理。
