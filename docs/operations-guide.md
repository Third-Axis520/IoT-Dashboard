# IoT Dashboard 系統操作指南

## 系統架構概覽

系統採用**全 UI 操作、DB 儲存**的彈性設計。所有產線、設備、感測器設定皆透過介面完成，不需修改程式碼。

| 資料類型 | 儲存位置 | 修改方式 |
|---------|---------|---------|
| 產線 / 設備配置 | 後端 DB | UI 操作 |
| 感測器屬性類型 | 後端 DB | UI → 屬性類型 |
| 設備連線（PLC/API） | 後端 DB | 設備整合精靈 |
| UCL / LCL 限值 | 後端 DB | UI → 限值設定 |
| 鞋子在位偵測 | 後端 DB | 綁定 `material_detect` 屬性 |

> **注意：** 目前前端仍保留一個舊的「烤箱生產線」Tab（硬編碼在 `liveLineConfig.ts`）。
> 正確做法是透過本指南的流程，用 UI 重新建立它，之後再從程式碼移除舊 Tab。

## 第一部分：建立新產線

### 1-1 新增產線

1. 點左上角**產線下拉選單**
2. 滾到底部 → **＋ 添加产线**
3. 輸入產線名稱（例如「烤箱生產線」）→ 按 Enter

### 1-2 刪除產線

1. 開啟**編輯模式**（右上角鎖頭圖示）
2. 點產線下拉 → 滑鼠移到產線名稱 → 出現垃圾桶圖示 → 點擊
3. 確認對話框會顯示產線名稱 → 點「刪除」確認

## 第二部分：新增設備到產線（設備整合精靈）

每台真實 PLC 設備都透過**設備整合精靈**建立，共 7 步。

### 開啟精靈

右上角「＋ 新增設備」下拉選單提供兩個選項：

| 選項 | 用途 | 步驟數 |
|------|------|--------|
| **整合新設備** | 從零開始：選協議、掃描、建立設備類型 | 7 步 |
| **使用已有模板** | 從已建立的設備類型新增卡片到產線 | 3 步 |

選擇「整合新設備」進入精靈：

### Step 1：選擇通訊協議

| 協議 | 適用情境 |
|------|---------|
| **Modbus TCP** | 標準 PLC（西門子、三菱、台達等） |
| **Web API** | 設備提供 REST API |
| **Push Ingest** | 設備主動推送（MQTT gateway 等） |

### Step 2：填入連線設定

**Modbus TCP：**
```
主機位址：設備 IP（例如 192.168.1.100）
Port：502
單元 ID：1（大多數設備預設）
```

**Web API：**
```
端點 URL：http://設備IP/api/data
HTTP 方法：GET
JSON 路徑：data.value（點號路徑）
輪詢間隔：5000（毫秒）
```

### Step 3：掃描設備資料點

點 **掃描設備** → 系統列出所有可用暫存器 → 勾選要監控的點。

### Step 4：選取資料點

從掃描結果中勾選要監控的暫存器。

### Step 5：貼屬性標籤

每個選取的資料點設定：
- **屬性類型**：溫度 / 壓力 / 濕度 / 流量 / 計數器 / 狀態 / 在位
- **顯示名稱**：自訂（例如「大底溫度」）
- **單位**：℃ / kPa / % 等

### Step 6：設定設備類型

```
設備類型名稱：例如「藥水箱」
顯示類型：
  single_kpi      — 單一大數字（1 個感測點）
  dual_side_spark — 左右兩組（2-3 個感測點）
  four_rings      — 四個環形指針（4 個感測點）
  molding_matrix  — 矩陣格式（6 個感測點）
```

### Step 7：確認建立

點 **建立** → 系統自動：
1. 建立 DeviceConnection（開始 Polling）
2. 建立 EquipmentType（設備類型範本）
3. SSE 推送更新到所有瀏覽器

### 精靈完成後：加入產線

精靈建立的是「設備類型範本」，還需手動加入產線：

1. 右上角下拉 → **加入既有類型**
2. 選取剛建立的設備類型
3. 完成 → 設備卡片出現在儀表板

| **40004** | 4 | 一次膠 | 大底溫度 | 80 | 50 | ℃ |
| **40005** | 5 | 一次膠 | 鞋面溫度 | 75 | 45 | ℃ |
| **40006** | 6 | 二次膠 | 大底溫度 | 80 | 50 | ℃ |
| **40007** | 7 | 二次膠 | 鞋面溫度 | 75 | 45 | ℃ |
| **40008** | 8 | 冷凍機 | 設備溫度 | -10 | -30 | ℃ |

### 鞋子在位偵測（DI 數位輸入）

原系統使用 **32 個 DI 數位輸入**（Modbus FC02 Discrete Input，從 address `0` 開始讀 32 個 bit）偵測各工位的鞋子在位狀態，並**非** register 40013。對應 `OvenDataReceive/Services/ModbusDataService.cs` 的 `ReadDiscreteInputs(unitId=1, startAddress=0, count=32)`。

> **重點：** DI 與溫度共用同一台 PLC（同 IP / Port / UnitID），但因為 Modbus 功能碼不同，必須**建兩條獨立的 DeviceConnection**（精靈跑兩次）。

---

#### Modbus 功能碼是什麼？該怎麼選？

精靈 Step 2 有「**Modbus 功能碼**」下拉，這個系統支援兩個值：

| 選項 | Modbus FC | 訊號類型 | 數值 | 用途 |
|------|-----------|---------|------|------|
| `holding` | **FC03** Holding Register | 類比 / 數字 | 16-bit 整數 | 溫度、壓力、濕度、流量、轉速 |
| `discrete` | **FC02** Discrete Input | 數位 ON/OFF | 1-bit (0/1) | 光電開關、限位、**鞋子到位偵測**、急停 |

選 `discrete` 時 `dataType` / `scale` / `byteSwap` 會被忽略，回值固定 `0.0` 或 `1.0`。

---

#### 步驟 — 建立 DI 連線（精靈跑第二次）

跑完上面 6 條溫度設備之後，再跑一次精靈專門建 DI：

| 精靈步驟 | 填入值 |
|---------|--------|
| Step 1 協議 | **Modbus TCP** |
| Step 2 主機位址 | `192.168.62.74`（與溫度同 IP）|
| Step 2 Port | `502` |
| Step 2 Unit ID | `1` |
| Step 2 起始位址 | `0`（DI 從 0 開始，不是 40001） |
| Step 2 讀取數量 | `32`（共 32 路 DI，FC02 最大 2000） |
| Step 2 **Modbus 功能碼** | **`discrete`** ← 重點 |
| Step 2 連線名稱 | `烤箱-DI` |
| Step 3 掃描 | 掃描後出現 32 個 bit 點位（DI01 ~ DI32） |
| Step 4 選取 | 勾選實際接到光電 / 到位開關的那幾個 bit |
| Step 5 屬性類型 | **「在位」**（material_detect） |
| Step 6 設備類型 | 例如 `烤箱DI集中器`，顯示類型 `single_kpi` |
| Step 7 建立 | 完成 |

---

#### Sensor Gating — 用 DI 控制溫度採樣

當你希望「**只有鞋子在工位上時才記錄溫度**」（避免空轉時段污染統計），用 sensor gating：

1. 上面兩個連線都建好
2. 開**溫度設備**那條 AssetCode 的 **🎚 限值設定**（LimitsSettingsModal）
3. 對要 gating 的溫度感測器（例如 `40003 鞋面溫度`）展開「**條件採樣**」
4. **gating 來源** 下拉 → 選 `烤箱-DI` 連線下對應的 DI bit（**支援跨 AssetCode**）
5. 設定 `DelayMs`（預設 0，gating 升起後多久開始採樣）和 `MaxAgeMs`（預設 1000，DI 訊號過期視為未到位）
6. 儲存 → 下一個 polling tick 立刻生效

效果：
- DI=1 → 溫度照常記錄（dashboard 綠色「採樣中」）
- DI=0 → 溫度**完全不寫 DB**（dashboard 灰色「待機」）
- DI 訊號超過 `MaxAgeMs` 沒更新 → **fail-safe 視為未到位**（dashboard 紅色「異常」）

詳細設計：`docs/superpowers/specs/2026-04-27-sensor-gating-design.md`

### 操作流程

1. 左上角產線下拉 → ＋ → 輸入「烤箱生產線」→ Enter
2. 右上角下拉 → **整合新設備** → 跑精靈（共 6 次，每台一次）— **溫度連線**
   - Step 1：Modbus TCP
   - Step 2：主機 `192.168.62.74`、Port `502`、Unit ID `1`、**Modbus 功能碼 `holding`**
   - Step 3：掃描後依上表選取對應 register（40001-40012）
   - Step 5：依上表設定屬性類型 + 顯示名稱
   - Step 6：填設備類型名稱 + 顯示類型（依上表）
   - Step 7：建立
3. 再跑一次精靈 — **DI 連線（鞋子在位偵測）**
   - Step 1：Modbus TCP
   - Step 2：主機 `192.168.62.74`、Port `502`、Unit ID `1`、起始位址 `0`、讀取數量 `32`、**Modbus 功能碼 `discrete`**
   - Step 3 ~ 7：依「鞋子在位偵測」段落
4. 精靈完成後 → **加入既有類型** → 選剛建的設備 → 加入烤箱生產線
5. 加入時設備綁定 AssetCode：`OVEN-LINE-01`
6. 重複步驟 2-5，直到 6 台溫度設備 + 1 條 DI 全部加入
7. 右上角 🎚 → **限值設定** → 依各設備表格填入 UCL / LCL → 視需要展開「條件採樣」綁定 DI gating 來源 → 儲存

## 第五部分：屬性類型管理

### 系統內建 8 個屬性類型（不可刪除）

| 名稱 | Behavior | 預設單位 | 說明 |
|------|---------|---------|------|
| 溫度 | normal | ℃ | 一般量測值 |
| 壓力 | normal | kPa | 一般量測值 |
| 濕度 | normal | % | 一般量測值 |
| 流量 | normal | L/min | 一般量測值 |
| 計數器 | counter | count | 累計計數 |
| 狀態 | state | — | 開關狀態 |
| 資產編號 | asset_code | — | 設備識別碼 |
| 在位 | material_detect | — | 鞋子在位偵測 |

> **在位偵測說明：** 只要某個感測點綁定了「在位」屬性類型，系統自動識別為在位感測器（原 PLC register 40013），不需硬編碼 SensorId。

### 新增自訂屬性類型

右上角 **⚙️ 系統設定** → **屬性管理** → **新增** → 填入名稱、圖示、預設單位、UCL/LCL 預設值

新增後在精靈 Step 5 的下拉清單中出現。

## 第六部分：系統監控

### 查看設備連線健康狀態

右上角 **⚙️ 系統設定** → **連線管理** → 顯示所有 DeviceConnection 狀態

或直接查 API：
```
GET /api/diagnostics/polling
```

回傳每個設備的：連線狀態（healthy / error / disabled）、連續錯誤次數、最後 Poll 時間、錯誤訊息

### Circuit Breaker 機制

- 連續 **3 次** Poll 失敗 → 切換慢速重試（每 30 秒一次）
- 恢復成功後自動回到正常間隔

### SSE 即時推送

設備連線、屬性類型等設定變更 → 自動推送到所有已開啟的瀏覽器，無需手動重新整理。
